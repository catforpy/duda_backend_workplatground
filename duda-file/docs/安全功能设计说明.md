## 1. 服务端加密

### 1.1 概述

OSS支持在服务器端对上传的数据进行加密,当数据通过HTTPS协议上传到OSS时,OSS会对数据进行加密,并将加密后的数据持久化存储。下载时自动解密返回给用户。

### 1.2 加密方式

#### 1.2.1 SSE-OSS (OSS托管密钥)

**特点**:
- 使用OSS完全托管的密钥进行加密
- 每个对象使用唯一的密钥
- 密钥由OSS管理,定期轮换
- **免费功能**,无额外费用

**适用场景**:
- 一般业务数据加密
- 不需要自定义密钥管理的场景
- 成本敏感型应用

**配置方式**:

```java
// 方式一:创建Bucket时设置默认加密
public void createBucketWithEncryption() {
    // 设置服务器端加密规则
    ServerSideEncryptionByDefault sseConfig = ServerSideEncryptionByDefault.builder()
        .sseAlgorithm(SSEAlgorithm.OSS_SERVICE_ENCRYPTION) // SSE-OSS
        .kmsMasterKeyID(null) // SSE-OSS不需要KMS密钥ID
        .build();

    PutBucketEncryptionRequest encryptionRequest = PutBucketEncryptionRequest.builder()
        .bucket("your-bucket-name")
        .serverSideEncryptionByDefault(sseConfig)
        .build();

    ossClient.putBucketEncryption(encryptionRequest);
}

// 方式二:上传对象时指定加密
public void uploadObjectWithSSEOSS() {
    ObjectMetadata metadata = new ObjectMetadata();
    metadata.setContentLength(1024);
    // 设置SSE-OSS加密
    metadata.setServerSideEncryption(ObjectMetadata.AES_256_SERVER_SIDE_ENCRYPTION);

    ossClient.putObject("your-bucket-name", "encrypted-file.txt",
                        new ByteArrayInputStream("test content".getBytes()),
                        metadata);
}
```

#### 1.2.2 SSE-KMS (KMS托管密钥)

**特点**:
- 使用密钥管理服务(KMS)托管的密钥
- 支持BYOK(Bring Your Own Key),自带密钥
- 可使用默认密钥(CSS服务默认服务密钥)或自定义密钥
- 提供密钥策略,精细控制访问权限
- **产生KMS费用**

**适用场景**:
- 需要自定义密钥管理的企业
- 需要BYOK合规要求的场景
- 需要细粒度密钥访问控制的场景
- 金融、政府等高安全要求行业

**配置方式**:

```java
// 方式一:创建Bucket时设置默认加密
public void createBucketWithSSEKMS() {
    ServerSideEncryptionByDefault sseConfig = ServerSideEncryptionByDefault.builder()
        .sseAlgorithm(SSEAlgorithm.KMS) // SSE-KMS
        .kmsMasterKeyID("your-kms-key-id") // 指定KMS密钥ID,不指定则使用默认密钥
        .build();

    PutBucketEncryptionRequest encryptionRequest = PutBucketEncryptionRequest.builder()
        .bucket("your-bucket-name")
        .serverSideEncryptionByDefault(sseConfig)
        .build();

    ossClient.putBucketEncryption(encryptionRequest);
}

// 方式二:上传对象时指定KMS加密
public void uploadObjectWithSSEKMS() {
    ObjectMetadata metadata = new ObjectMetadata();
    metadata.setContentLength(1024);
    // 设置SSE-KMS加密,指定KMS密钥ID
    metadata.setServerSideEncryptionKeyId("your-kms-key-id");

    ossClient.putObject("your-bucket-name", "encrypted-file.txt",
                        new ByteArrayInputStream("test content".getBytes()),
                        metadata);
}
```

### 1.3 加密策略

#### 1.3.1 Bucket级别加密

为整个Bucket设置默认加密策略,所有上传到该Bucket的对象都会自动加密:

```java
/**
 * 设置Bucket默认加密策略
 */
public void setBucketEncryption(String bucketName, EncryptionType encryptionType, String kmsKeyId) {
    ServerSideEncryptionByDefault sseConfig;

    if (encryptionType == EncryptionType.SSE_OSS) {
        sseConfig = ServerSideEncryptionByDefault.builder()
            .sseAlgorithm(SSEAlgorithm.OSS_SERVICE_ENCRYPTION)
            .build();
    } else if (encryptionType == EncryptionType.SSE_KMS) {
        sseConfig = ServerSideEncryptionByDefault.builder()
            .sseAlgorithm(SSEAlgorithm.KMS)
            .kmsMasterKeyID(kmsKeyId) // 可为null,使用默认KMS密钥
            .build();
    } else {
        throw new IllegalArgumentException("Unsupported encryption type");
    }

    PutBucketEncryptionRequest request = PutBucketEncryptionRequest.builder()
        .bucket(bucketName)
        .serverSideEncryptionByDefault(sseConfig)
        .build();

    ossClient.putBucketEncryption(request);
}

/**
 * 获取Bucket加密配置
 */
public BucketEncryptionConfiguration getBucketEncryption(String bucketName) {
    GetBucketEncryptionRequest request = GetBucketEncryptionRequest.builder()
        .bucket(bucketName)
        .build();

    return ossClient.getBucketEncryption(request);
}

/**
 * 删除Bucket加密配置
 */
public void deleteBucketEncryption(String bucketName) {
    DeleteBucketEncryptionRequest request = DeleteBucketEncryptionRequest.builder()
        .bucket(bucketName)
        .build();

    ossClient.deleteBucketEncryption(request);
}
```

#### 1.3.2 对象级别加密

单个对象上传时指定加密方式,优先级高于Bucket默认加密:

```java
/**
 * 上传文件并指定加密方式
 */
public PutObjectResult uploadObjectWithEncryption(
    String bucketName,
    String objectKey,
    InputStream inputStream,
    ObjectMetadata metadata,
    EncryptionConfig encryptionConfig
) {
    if (encryptionConfig.getType() == EncryptionType.SSE_OSS) {
        metadata.setServerSideEncryption(ObjectMetadata.AES_256_SERVER_SIDE_ENCRYPTION);
    } else if (encryptionConfig.getType() == EncryptionType.SSE_KMS) {
        metadata.setServerSideEncryptionKeyId(encryptionConfig.getKmsKeyId());
    }

    return ossClient.putObject(bucketName, objectKey, inputStream, metadata);
}
```

### 1.5 加密策略对比

| 特性 | SSE-OSS | SSE-KMS |
|------|---------|---------|

| **密钥管理** | OSS托管 | KMS托管 |
| **BYOK支持** | 不支持 | 支持 |
| **密钥轮换** | 自动 | 自动/手动 |
| **访问控制** | Bucket级别 | 密钥级别,更细粒度 |
| **适用场景** | 一般业务 | 企业合规、高安全要求 |
| **性能影响** | 极小 | 小,需调用KMS API |

## 2. 客户端加密

### 2.1 概述

客户端加密是指数据在本地加密后再上传到OSS,下载时需要先下载加密数据,然后在本地解密。这种方式确保只有持有密钥的客户端才能访问数据。

### 2.2 客户端加密流程

**上传流程**:
1. 客户端生成对称密钥(数据密钥)
2. 使用数据密钥加密明文数据
3. 使用主密钥(RSA或KMS)加密数据密钥
4. 上传加密后的数据和加密后的数据密钥

**下载流程**:
1. 下载加密数据和加密的数据密钥
2. 使用主密钥解密数据密钥
3. 使用数据密钥解密数据

### 2.3 实现方式

```java
/**
 * 客户端加密示例
 */
public class ClientSideEncryptionExample {

    private OSS ossClient;
    private String bucketName = "your-bucket-name";

    // 方式一:使用RSA主密钥
    public void encryptWithRSA() throws Exception {
        // 1. 生成RSA密钥对
        KeyPairGenerator keyPairGenerator = KeyPairGenerator.getInstance("RSA");
        keyPairGenerator.initialize(2048);
        KeyPair keyPair = keyPairGenerator.generateKeyPair();

        // 2. 创建客户端加密配置
        Configuration config = new Configuration();
        config.setSignatureVersion(com.aliyun.oss.common.auth.SignVersion.V4);

        // 3. 使用RSA主密钥进行客户端加密
        CryptoConfiguration cryptoConfig = new CryptoConfiguration();
        cryptoConfig.setCryptoMode(CryptoMode.AuthenticatedEncryption); // 认证加密模式

        KMSEncryptionMaterials encryptionMaterials = new KMSEncryptionMaterials(
            "your-kms-region", "your-kms-key-id");

        OSSCryptoClient cryptoClient = new OSSCryptoClient(ossClient, encryptionMaterials, cryptoConfig);

        // 4. 上传文件(自动加密)
        InputStream inputStream = new FileInputStream("plain-file.txt");
        cryptoClient.putObject(bucketName, "encrypted-file.txt", inputStream);
    }

    // 方式二:使用KMS主密钥
    public void encryptWithKMS() {
        // 1. 创建KMS加密材料
        KMSEncryptionMaterials encryptionMaterials = new KMSEncryptionMaterials(
            "your-kms-region",
            "your-kms-key-id" // 可为null,使用默认KMS密钥
        );

        // 2. 创建加密配置
        CryptoConfiguration cryptoConfig = new CryptoConfiguration();
        cryptoConfig.setCryptoMode(CryptoMode.AuthenticatedEncryption);

        // 3. 创建加密客户端
        OSSCryptoClient cryptoClient = new OSSCryptoClient(ossClient, encryptionMaterials, cryptoConfig);

        // 4. 上传文件(自动加密)
        InputStream inputStream = new FileInputStream("plain-file.txt");
        cryptoClient.putObject(bucketName, "encrypted-file.txt", inputStream);
    }

    // 方式三:使用对称主密钥
    public void encryptWithSymmetricKey() throws Exception {
        // 1. 生成对称密钥
        KeyGenerator keyGenerator = KeyGenerator.getInstance("AES");
        keyGenerator.init(256);
        SecretKey secretKey = keyGenerator.generateKey();

        // 2. 创建对称加密材料
        SymmetricKeyEncryptionMaterials encryptionMaterials = new SymmetricKeyEncryptionMaterials(secretKey);

        // 3. 创建加密配置
        CryptoConfiguration cryptoConfig = new CryptoConfiguration();
        cryptoConfig.setCryptoMode(CryptoMode.AuthenticatedEncryption);

        // 4. 创建加密客户端
        OSSCryptoClient cryptoClient = new OSSCryptoClient(ossClient, encryptionMaterials, cryptoConfig);

        // 5. 上传文件(自动加密)
        InputStream inputStream = new FileInputStream("plain-file.txt");
        cryptoClient.putObject(bucketName, "encrypted-file.txt", inputStream);
    }
}
```

### 2.4 客户端加密模式

| 加密模式 | 说明 | 安全性 | 性能 | 推荐场景 |
|----------|------|--------|------|----------|
| **AuthenticatedEncryption** | 认证加密模式(AES-GCM) | 高 | 中 | 高安全要求场景 |
| **EncryptionOnly** | 仅加密模式(AES-CBC) | 中 | 高 | 性能敏感场景 |

### 2.5 注意事项

1. **密钥管理**: 客户端主密钥必须妥善保管,丢失后无法解密数据
2. **兼容性**: 客户端加密的数据只能通过加密SDK下载,普通OSS API无法正确读取
3. **元数据加密**: 客户端加密会添加加密相关元数据,对象元数据不能存储敏感信息
4. **跨平台**: 不同平台的加密SDK可能不兼容,需要统一加密库版本

## 3. TLS版本配置

### 3.1 概述

OSS支持通过HTTPS协议进行安全传输,支持TLS 1.0、1.1、1.2、1.3等多个版本。不同版本的TLS在安全性和兼容性上有差异。

### 3.2 TLS版本对比

| TLS版本 | 发布年份 | 安全性 | 浏览器兼容性 | 推荐使用 |
|---------|----------|--------|--------------|----------|
| **TLS 1.0** | 1999 | 低,存在已知漏洞 | 所有浏览器 | ❌ 不推荐 |
| **TLS 1.1** | 2006 | 中 | 所有浏览器 | ❌ 不推荐 |
| **TLS 1.2** | 2008 | 高 | 主流浏览器 | ✅ 推荐 |
| **TLS 1.3** | 2018 | 很高 | 现代浏览器 | ✅ 强烈推荐 |

### 3.3 加密套件配置

```java
/**
 * 配置TLS加密套件
 */
public class TLSConfiguration {

    // 方式一:使用Java系统属性配置
    public void configureTLSWithSystemProperties() {
        // 启用TLSv1.2和TLSv1.3
        System.setProperty("https.protocols", "TLSv1.2,TLSv1.3");

        // 配置加密套件
        System.setProperty("https.cipherSuites",
            "TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384," +
            "TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384," +
            "TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256," +
            "TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256"
        );
    }

    // 方式二:在OSS Client中配置
    public OSSClient createOSSClientWithTLS() {
        // 创建ClientConfiguration
        ClientBuilderConfiguration config = new ClientBuilderConfiguration();

        // 设置支持的TLS版本
        config.setProtocol(Protocol.HTTPS); // 使用HTTPS

        // 设置请求超时
        config.setRequestTimeout(30000);
        config.setConnectionTimeout(10000);

        // 创建OSSClient
        return new OSSClientBuilder().build(
            "your-endpoint",
            "your-access-key-id",
            "your-access-key-secret",
            config
        );
    }
}
```

### 3.4 最佳实践

1. **优先使用TLS 1.3**: TLS 1.3提供最强的安全性,握手延迟更低
2. **禁用旧版本**: 生产环境应禁用TLS 1.0和1.1
3. **使用强加密套件**: 优先选择AES-GCM和ChaCha20-Poly1305等AEAD加密套件
4. **定期更新**: 及时更新SSL/TLS库,修复已知漏洞
5. **证书验证**: 始终启用服务器证书验证,防止中间人攻击

## 4. 数据完整性校验

### 4.1 概述

OSS支持通过MD5和CRC64两种算法校验数据的完整性,确保数据在上传、下载、传输过程中未被篡改或损坏。

### 4.2 MD5校验

#### 4.2.1 上传时MD5校验

```java
/**
 * 上传文件并进行MD5校验
 */
public void uploadWithMD5Check(String bucketName, String filePath, String objectKey) throws Exception {
    File file = new File(filePath);

    // 1. 计算文件的MD5值
    String contentMD5 = calculateMD5(file);

    // 2. 设置对象元数据
    ObjectMetadata metadata = new ObjectMetadata();
    metadata.setContentLength(file.length());
    metadata.setContentMD5(contentMD5); // OSS会根据MD5值校验数据完整性

    // 3. 上传文件
    PutObjectResult result = ossClient.putObject(bucketName, objectKey,
                                                 new FileInputStream(file),
                                                 metadata);

    // 4. 验证ETag(对于简单上传,ETag即为MD5值)
    String etag = result.getETag();
    System.out.println("Upload ETag: " + etag);
}

/**
 * 计算文件的MD5值
 */
private String calculateMD5(File file) throws Exception {
    MessageDigest md = MessageDigest.getInstance("MD5");
    try (FileInputStream fis = new FileInputStream(file);
         DigestInputStream dis = new DigestInputStream(fis, md)) {
        // 读取文件内容
        byte[] buffer = new byte[8192];
        while (dis.read(buffer) != -1) {
            // 继续读取
        }
    }

    // 获取MD5值并进行Base64编码
    byte[] digest = md.digest();
    return BinaryUtil.toBase64String(digest);
}
```

#### 4.2.2 下载时MD5校验

```java
/**
 * 下载文件并验证MD5
 */
public void downloadWithMD5Check(String bucketName, String objectKey, String localPath) throws Exception {
    // 1. 获取对象元数据
    OSSObject ossObject = ossClient.getObject(bucketName, objectKey);
    ObjectMetadata metadata = ossObject.getObjectMetadata();

    // 2. 读取服务器返回的Content-MD5
    String serverMD5 = metadata.getContentMD5();

    // 3. 计算本地文件的MD5
    MessageDigest md = MessageDigest.getInstance("MD5");
    try (InputStream inputStream = ossObject.getObjectContent();
         FileOutputStream fos = new FileOutputStream(localPath)) {

        byte[] buffer = new byte[8192];
        int bytesRead;
        while ((bytesRead = inputStream.read(buffer)) != -1) {
            fos.write(buffer, 0, bytesRead);
            md.update(buffer, 0, bytesRead);
        }
    }

    // 4. 计算本地MD5值
    byte[] digest = md.digest();
    String localMD5 = BinaryUtil.toBase64String(digest);

    // 5. 比较MD5值
    if (!serverMD5.equals(localMD5)) {
        throw new IOException("MD5 check failed! Data may be corrupted.");
    }

    System.out.println("MD5 check passed! File downloaded successfully.");
}
```

### 4.3 CRC64校验

#### 4.3.1 概述

CRC64是一种循环冗余校验算法,相比MD5具有以下优势:
- 计算速度快,性能影响小
- OSS支持开启CRC64自动校验
- 适用于大文件上传下载

#### 4.3.2 开启CRC64校验

```java
/**
 * 开启CRC64校验
 */
public void enableCRC64Check() {
    // 创建ClientConfiguration并启用CRC64校验
    ClientBuilderConfiguration config = new ClientBuilderConfiguration();
    config.setCRC64Enabled(true); // 开启CRC64校验

    // 创建OSSClient
    OSSClient ossClient = new OSSClientBuilder().build(
        "your-endpoint",
        "your-access-key-id",
        "your-access-key-secret",
        config
    );
}
```

#### 4.3.3 上传时CRC64校验

```java
/**
 * 上传文件并使用CRC64校验
 */
public void uploadWithCRC64Check(String bucketName, String filePath, String objectKey) throws Exception {
    File file = new File(filePath);

    // 1. 计算文件的CRC64值
    long clientCRC64 = calculateCRC64(file);

    // 2. 设置对象元数据
    ObjectMetadata metadata = new ObjectMetadata();
    metadata.setContentLength(file.length());

    // 3. 上传文件
    PutObjectResult result = ossClient.putObject(bucketName, objectKey,
                                                 new FileInputStream(file),
                                                 metadata);

    // 4. 获取服务器返回的CRC64值
    Long serverCRC64 = result.getClientCRC64();
    System.out.println("Client CRC64: " + clientCRC64);
    System.out.println("Server CRC64: " + serverCRC64);
}

/**
 * 计算文件的CRC64值
 */
private long calculateCRC64(File file) throws Exception {
    CRC64 crc64 = new CRC64();

    try (FileInputStream fis = new FileInputStream(file)) {
        byte[] buffer = new byte[8192];
        int bytesRead;
        while ((bytesRead = fis.read(buffer)) != -1) {
            crc64.update(buffer, 0, bytesRead);
        }
    }

    return crc64.getValue();
}
```

#### 4.3.4 下载时CRC64校验

```java
/**
 * 下载文件并验证CRC64
 */
public void downloadWithCRC64Check(String bucketName, String objectKey, String localPath) throws Exception {
    // 1. 获取对象元数据
    OSSObject ossObject = ossClient.getObject(bucketName, objectKey);
    ObjectMetadata metadata = ossObject.getObjectMetadata();

    // 2. 计算本地文件的CRC64值
    CRC64 crc64 = new CRC64();
    try (InputStream inputStream = ossObject.getObjectContent();
         FileOutputStream fos = new FileOutputStream(localPath)) {

        byte[] buffer = new byte[8192];
        int bytesRead;
        while ((bytesRead = inputStream.read(buffer)) != -1) {
            fos.write(buffer, 0, bytesRead);
            crc64.update(buffer, 0, bytesRead);
        }
    }

    // 3. 获取服务器返回的CRC64值
    Long serverCRC64 = metadata.getServerCRC64();
    long localCRC64 = crc64.getValue();

    // 4. 比较CRC64值
    if (serverCRC64 != null && serverCRC64 != localCRC64) {
        throw new IOException("CRC64 check failed! Data may be corrupted.");
    }

    System.out.println("CRC64 check passed! File downloaded successfully.");
}
```

### 4.4 MD5 vs CRC64对比

| 特性 | MD5 | CRC64 |
|------|-----|-------|
| **计算速度** | 较慢 | 快 |
| **冲突概率** | 极低 | 低 |
| **安全性** | 密码学哈希,可用于签名 | 非密码学校验,仅用于完整性校验 |
| **适用场景** | 小文件、需要安全校验的场景 | 大文件、性能敏感场景 |
| **自动校验** | 需手动设置 | OSS客户端可自动校验 |

### 4.5 校验策略选择

```java
/**
 * 完整性校验策略
 */
public enum IntegrityCheckType {
    NONE,           // 不进行完整性校验
    MD5,            // 使用MD5校验
    CRC64,          // 使用CRC64校验
    BOTH            // 同时使用MD5和CRC64校验
}

/**
 * 文件上传配置
 */
@Data
@Builder
public class UploadConfig {
    private IntegrityCheckType checkType;  // 完整性校验类型
    private Boolean enableCRC64;           // 是否启用CRC64自动校验

    /**
     * 根据文件大小选择合适的校验策略
     */
    public static UploadConfig getRecommendedConfig(long fileSize) {
        // 小文件(< 100MB):使用MD5校验
        if (fileSize < 100 * 1024 * 1024) {
            return UploadConfig.builder()
                .checkType(IntegrityCheckType.MD5)
                .enableCRC64(false)
                .build();
        }
        // 大文件(>= 100MB):使用CRC64校验
        else {
            return UploadConfig.builder()
                .checkType(IntegrityCheckType.CRC64)
                .enableCRC64(true)
                .build();
        }
    }
}
```

## 10. 配置管理

### 10.1 安全功能配置

在`application.yml`中配置安全功能:

```yaml
# 安全功能配置
duda:
  file:
    security:
      # 加密配置
      encryption:
        enabled: true                    # 是否启用加密
        default-type: SSE_OSS            # 默认加密类型: SSE_OSS, SSE_KMS
        default-kms-key-id: null         # 默认KMS密钥ID
        kms-region: cn-hangzhou          # KMS区域

      # TLS配置
      tls:
        version: TLSv1.3                 # TLS版本: TLSv1.0, TLSv1.1, TLSv1.2, TLSv1.3
        cipher-suites:                   # 加密套件
          - TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384
          - TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384

      # 完整性校验配置
      integrity:
        enabled: true                    # 是否启用完整性校验
        check-type: CRC64                # 校验类型: NONE, MD5, CRC64, BOTH
        auto-check: true                 # 是否自动校验

      # 内容安全检测配置
      content-detection:
        enabled: false                   # 是否启用内容安全检测(默认关闭)
        auto-detect: false               # 是否自动检测新上传文件
        threshold: 90.00                 # 检测阈值(0-100)
        auto-reject-high-risk: true      # 是否自动拒绝高风险内容

      # 病毒检测配置
      virus-scan:
        enabled: false                   # 是否启用病毒检测(默认关闭)
        auto-scan: false                 # 是否自动扫描新上传文件
        action: DELETE                   # 发现病毒后的操作: DELETE, QUARANTINE, MARK
        max-file-size: 524288000         # 最大文件大小(500MB)

      # 敏感数据扫描配置
      sensitive-data:
        enabled: false                   # 是否启用敏感数据扫描(默认关闭)
        auto-scan: false                 # 是否自动扫描新上传文件
        scan-levels:                     # 扫描等级
          - S2
          - S3
          - S4
        max-file-size: 209715200         # 最大文件大小(200MB)
```

## 11. 实施建议

### 11.1 分阶段实施

**第一阶段(基础安全)**:
1. 启用服务端加密(SSE-OSS)
2. 配置TLS 1.2/1.3
3. 启用CRC64完整性校验

**第二阶段(内容安全)**:
1. 启用内容安全检测
2. 配置自动拦截规则
3. 实施人工审核流程

**第三阶段(高级安全)**:
1. 启用病毒检测
2. 启用敏感数据扫描
3. 实施数据脱敏和加密

### 11.2 成本控制

1. **按需启用**: 根据实际需求启用安全功能,避免不必要的费用
2. **批量处理**: 内容检测和病毒扫描支持批量操作,可降低成本
3. **缓存结果**: 检测结果可缓存,避免重复检测

### 11.3 性能优化

1. **异步处理**: 内容检测和病毒扫描建议异步处理
2. **MQ通知**: 使用MQ通知检测结果,避免轮询
3. **分批扫描**: 大规模扫描任务分批进行,避免影响业务

### 11.4 合规要求

1. **金融行业**: 必须启用SSE-KMS加密和敏感数据扫描
2. **医疗行业**: 必须启用敏感数据扫描和数据脱敏
3. **政府行业**: 必须启用内容安全检测和病毒检测
4. **一般业务**: 推荐启用基础加密和完整性校验

## 12. 总结

本文档详细说明了duda-file模块的安全功能设计,包括:

1. **服务端加密**: 支持SSE-OSS和SSE-KMS两种加密方式
2. **客户端加密**: 提供多种客户端加密模式
3. **TLS版本配置**: 支持TLS 1.0-1.3,推荐使用TLS 1.2/1.3
4. **数据完整性校验**: 支持MD5和CRC64两种校验算法
5. **内容安全检测**: 检测图片中的违规内容
6. **病毒检测**: 检测文件中的恶意代码和病毒
7. **敏感数据保护**: 识别和保护敏感数据

这些安全功能可以根据实际需求选择性启用,建议分阶段实施,在保障数据安全的同时控制成本。

**重要提示**: 这些安全功能为后期可能增加的功能,短期内可能不会实现。在实施前需要充分评估业务需求、成本预算和性能影响。
