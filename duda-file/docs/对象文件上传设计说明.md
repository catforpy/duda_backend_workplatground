# 对象文件上传设计说明

## 概述

基于阿里云OSS的对象上传文档,本文档整理了文件上传的核心概念、命名规范、上传方式及适用场景,为后续统一设计提供基础。

## 一、Object基本概念

### 1.1 Object组成

**Object**是对象存储中数据的基本单位,类似于文件系统中的文件。

**核心组成部分**:

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| **Key** | Object名称,用于唯一标识Object | "exampledir/example.jpg" |
| **Data** | Object的实际数据,任意长度字节流 | 文件的二进制内容 |
| **Object Meta** | 元数据,键值对形式描述属性 | 文件大小、类型、最后修改时间 |
| **Version ID** | 版本标识,开启版本控制时分配 | "1158346987923434" |

### 1.2 Object命名规范（Key）

**命名规则**:
- 使用UTF-8编码
- 长度: 1~1023字节
- **不能**以正斜线(/)或反斜线(\)开头
- 区分大小写

**命名建议**:
1. 使用有意义的名称（文件名、日期、用户ID等）
2. 使用前缀组织数据,创建层次结构
3. 确保名称唯一性,可包含随机数或UUID

**命名示例**:

```
✅ 正确示例:
- avatar/user-123456-20260313.jpg
- documents/2026/03/report.pdf
- videos/category-a/video-001.mp4

❌ 错误示例:
- /exampleobject.txt (以/开头)
- \exampleobject.txt (以\开头)
- 太长的文件名超过1023字节.txt
```

**在Bucket中的位置表示**:

| 描述 | Key表示 |
|------|---------|
| 根目录下的exampleobject.txt | `exampleobject.txt` |
| destdir目录下的exampleobject.jpg | `destdir/exampleobject.jpg` |
| 多层级目录下的file.pdf | `documents/2026/03/file.pdf` |

### 1.3 OSS目录结构

**重要概念**: OSS本身**没有真实的文件夹概念**,而是通过Key中的"/"来模拟目录结构。

```
实际存储:
Key = "exampledir/example.jpg"

模拟效果:
exampledir/          (模拟的文件夹)
  └── example.jpg    (模拟的文件)
```

**目录管理操作**:
- 创建目录 = 创建一个以"/"结尾的Object (Key = "exampledir/")
- 删除目录 = 删除所有匹配该前缀的Object
- 列举目录 = 按前缀列举Object (Prefix = "exampledir/")

## 二、Object类型

### 2.1 Object类型分类

| Object类型 | 生成方式 | 特点 | 适用场景 |
|-----------|---------|------|---------|
| **Normal** | 简单上传(PutObject) | 内容固定,只能读取不能修改 | 普通文件上传 |
| **Multipart** | 分片上传 | 由多个分片合并而成 | 大文件上传 |
| **Appendable** | 追加上传(AppendObject) | 可在末尾追加内容 | 日志、实时视频流 |
| **Symlink** | 软链接(PutSymlink) | 指向其他Object的快捷方式 | 快速访问常用Object |
| **Delete Marker** | 删除标记(版本控制) | 标记Object已删除,可恢复 | 开启版本控制时 |

### 2.2 Object类型转换限制

**重要**: 不支持在不同类型的Object之间相互转换。

- Normal类型 → 无法转换为Multipart或Appendable
- Multipart类型 → 无法转换为Normal或Appendable
- Appendable类型 → 无法转换为Normal或Multipart

### 2.3 版本控制与Object覆盖

**场景**: 多用户上传同名文件

**未开启版本控制**:
```
用户A上传 example.jpg → 完成时间: 10:00:05
用户B上传 example.jpg → 完成时间: 10:00:03
最终结果: 保留用户A的文件 (后完成的上传覆盖先完成的)
```

**开启版本控制**:
```
用户A上传 example.jpg → 开始时间: 10:00:00 → Version ID: 1
用户B上传 example.jpg → 开始时间: 10:00:02 → Version ID: 2
最终结果: 用户B的文件标记为最新版本 (根据开始时间判断)
```

## 三、文件上传方式

### 3.1 上传方式总览

| 上传方式 | 文件大小限制 | 适用场景 | 优缺点 |
|---------|-------------|---------|--------|
| **简单上传** | ≤5GB | 小文件,一次HTTP请求完成 | ✅ 简单快速 ❌ 不适合大文件 |
| **分片上传** | ≤48.8TB | 大文件,需要断点续传 | ✅ 支持断点续传 ❌ 实现复杂 |
| **断点续传上传** | ≤48.8TB | 网络不稳定,大文件上传 | ✅ 自动断点续传 ❌ 需要Checkpoint文件 |
| **表单上传** | ≤5GB | Web端直接上传 | ✅ 减轻服务器压力 ❌ 不支持分片 |
| **追加上传** | ≤5GB | 日志、实时视频流 | ✅ 可追加内容 ❌ 不支持归档存储 |
| **预签名URL上传** | ≤5GB (单文件) | 第三方临时授权上传 | ✅ 安全灵活 ❌ URL有时效限制 |
| **STS临时凭证上传** | 不限 | 客户端直传 | ✅ 权限可控 ❌ 需要STS服务 |

### 3.2 简单上传（PutObject）

**适用场景**:
- 文件大小 ≤5GB
- 对并发上传速度要求不高
- 一次HTTP请求即可完成

**上传流程**:
```
客户端 → 选择文件 → 请求上传 → 上传到OSS → 返回结果
```

**核心代码逻辑**:
```java
// 创建上传请求
PutObjectRequest request = new PutObjectRequest(
    bucketName,      // Bucket名称
    objectKey,       // Object完整路径
    file             // 本地文件
);

// 设置元数据（可选）
ObjectMetadata metadata = new ObjectMetadata();
metadata.setContentType("image/jpeg");
metadata.setContentLength(file.length());

// 执行上传
PutObjectResult result = ossClient.putObject(request);
```

**可设置的参数**:
- **存储类型** (StorageClass): STANDARD, IA, ARCHIVE
- **访问权限** (ACL): PRIVATE, PUBLIC_READ, PUBLIC_READ_WRITE
- **用户元数据** (UserMetadata): 自定义键值对
- **服务器端加密** (ServerSideEncryption): AES256, KMS

**防止覆盖同名文件**:
- 方式1: 开启版本控制
- 方式2: 设置`x-oss-forbid-overwrite=true`,同名文件上传失败

**计费项**:
- PUT类型请求费用
- 存储费用（按大小和时长）

### 3.3 分片上传（MultipartUpload）

**适用场景**:
- 文件大小 >5GB
- 网络不稳定需要断点续传
- 需要提高上传速度（并发上传分片）

**工作原理**:
```
1. 初始化上传任务 → 获取Upload ID
   ↓
2. 切分文件为多个分片 → 并发上传
   ↓
3. 合并分片 → 完成上传
```

**分片规则**:
- 分片数量: 1~10,000个
- 单个分片大小: 100KB~5GB
- 最后一个分片可 <100KB

**上传流程**:

**步骤1: 初始化分片上传**
```java
InitiateMultipartUploadRequest request =
    new InitiateMultipartUploadRequest(bucketName, objectKey);

InitiateMultipartUploadResult result =
    ossClient.initiateMultipartUpload(request);

String uploadId = result.getUploadId(); // 保存此ID
```

**步骤2: 上传分片**
```java
List<PartETag> partETags = new ArrayList<>();
int partNumber = 1;

for (文件分片) {
    UploadPartRequest uploadPartRequest = new UploadPartRequest();
    uploadPartRequest.setBucketName(bucketName);
    uploadPartRequest.setKey(objectKey);
    uploadPartRequest.setUploadId(uploadId);
    uploadPartRequest.setPartNumber(partNumber);
    uploadPartRequest.setInputStream(partInputStream);

    UploadPartResult uploadPartResult = ossClient.uploadPart(uploadPartRequest);
    partETags.add(uploadPartResult.getPartETag());
    partNumber++;
}
```

**步骤3: 合并分片**
```java
CompleteMultipartUploadRequest completeRequest =
    new CompleteMultipartUploadRequest(bucketName, objectKey, uploadId, partETags);

CompleteMultipartUploadResult result =
    ossClient.completeMultipartUpload(completeRequest);
```

**取消上传**:
```java
// 中断上传任务,删除已上传的分片
ossClient.abortMultipartUpload(
    new AbortMultipartUploadRequest(bucketName, objectKey, uploadId)
);
```

**清理碎片文件**:
- **手动清理**: 控制台/ossbrowser
- **生命周期规则**: 自动清理过期碎片
- **API取消**: AbortMultipartUpload

**性能优化建议**:
1. **合理控制并发数**: 根据网络带宽调整
2. **避免顺序前缀命名**: 防止热点问题
3. **断点续传**: 失败后只需重传失败分片

**限制**:
| 限制项 | 限制值 |
|-------|--------|
| 单个文件大小 | ≤48.8TB |
| 分片数量 | 1~10,000个 |
| 单个分片大小 | 100KB~5GB (最后分片可<100KB) |
| ListParts返回数量 | ≤1,000个 |

### 3.4 断点续传上传

**适用场景**:
- 大文件上传(>5GB)
- 网络不稳定,容易中断
- 需要自动从断点继续

**核心特性**:
- 将文件分成多个小分片并发上传
- 使用**Checkpoint文件**记录上传进度
- 某分片失败后,只需重传该分片

**Checkpoint文件结构**:
```json
{
  "uploadId": "BE2D0BC931BE4DE1B23F339AABFA49EE",
  "bucketName": "examplebucket",
  "key": "exampleobject.jpg",
  "partSize": 1048576,
  "uploadedParts": [
    {"partNumber": 1, "etag": "\"xxx\"", "size": 1048576},
    {"partNumber": 2, "etag": "\"yyy\"", "size": 1048576}
  ]
}
```

**核心代码**:
```java
UploadFileRequest request = new UploadFileRequest(
    bucketName,
    objectKey,
    localFilePath
);

// 配置参数
request.setTaskNum(5);              // 并发线程数
request.setPartSize(1 * 1024 * 1024); // 分片大小(1MB)
request.setEnableCheckpoint(true);     // 启用断点续传
request.setCheckpointFile("checkpoint.ucp"); // Checkpoint文件路径

// 执行上传
ossClient.uploadFile(request);
```

**注意事项**:
1. 确保程序对Checkpoint文件有写权限
2. 不要修改Checkpoint文件中的校验信息
3. 本地文件改变会重新上传所有分片
4. 上传完成后Checkpoint文件会被删除

### 3.5 表单上传（PostObject）

**适用场景**:
- Web应用直接上传到OSS
- 不经过业务服务器中转
- 减轻服务器压力

**典型应用**:
- 用户上传头像
- 电商商品图片
- 简历、作品集上传
- 作业文件提交

**文件大小限制**: ≤5GB

**上传流程**:
```
1. 客户端请求上传参数
   ↓
2. 服务端生成Post签名和Post Policy
   ↓
3. 客户端使用签名上传文件到OSS
   ↓
4. OSS返回上传结果
```

**服务端生成签名**:
```java
// 设置过期时间
Date expiration = new Date(System.currentTimeMillis() + 3600 * 1000);

// 设置上传策略
PolicyConditions policyConds = new PolicyConditions();
policyConds.addConditionItem(PolicyConditions.COND_CONTENT_LENGTH_RANGE, 0, 1048576000);
policyConds.addConditionItem(MatchMode.StartWith, PolicyConditions.COND_KEY, "user-dir/");

// 生成Post Policy和签名
String postPolicy = ossClient.generatePostPolicy(expiration, policyConds);
String encodedPolicy = BinaryUtil.toBase64String(postPolicy.getBytes("utf-8"));
String postSignature = ossClient.calculatePostSignature(postPolicy);

// 返回给客户端
{
  "ossAccessKeyId": "LTAI***",
  "policy": "base64编码的policy",
  "signature": "签名值",
  "dir": "user-dir/",
  "host": "https://bucket.oss-cn-hangzhou.aliyuncs.com"
}
```

**客户端上传**:
```html
<form>
  <input type="file" id="file" name="file">
  <button type="submit">上传</button>
</form>

<script>
const form = document.querySelector('form');
form.addEventListener('submit', async (event) => {
  event.preventDefault();

  // 1. 获取上传参数
  const response = await fetch('/get_upload_params');
  const data = await response.json();

  // 2. 构建FormData
  const formData = new FormData();
  formData.append('name', filename);
  formData.append('policy', data.policy);
  formData.append('OSSAccessKeyId', data.ossAccessKeyId);
  formData.append('signature', data.signature);
  formData.append('key', data.dir + filename);
  formData.append('file', fileInput.files[0]);

  // 3. 上传到OSS
  await fetch(data.host, {
    method: 'POST',
    body: formData
  });
});
</script>
```

**Post Policy限制示例**:
```json
{
  "expiration": "2026-03-14T10:00:00Z",
  "conditions": [
    {"bucket": "examplebucket"},
    ["starts-with", "$key", "user/"],
    ["content-length-range", 0, 104857600]
  ]
}
```

**防止覆盖同名文件**:
- 携带参数`x-oss-forbid-overwrite=true`

### 3.6 追加上传（AppendObject）

**适用场景**:
- **视频监控**: 实时视频流上传
- **日志收集**: 持续追加日志内容
- **实时数据采集**: IoT设备数据上传

**核心特性**:
- 生成**Appendable类型**的Object
- 可在Object末尾直接追加内容
- 每次追加后数据**立即可读**
- 不支持冷归档、深度冷归档存储

**追加上传流程**:
```
第1次追加: position=0 → 创建Appendable Object
第2次追加: position=上一个Object长度
第3次追加: position=上一个Object长度
...
```

**核心代码**:
```java
String content1 = "Hello OSS A\n";
String content2 = "Hello OSS B\n";
String content3 = "Hello OSS C\n";

// 第1次追加 (创建文件)
AppendObjectRequest request1 = new AppendObjectRequest(
    bucketName,
    objectName,
    new ByteArrayInputStream(content1.getBytes())
);
request1.setPosition(0L); // 第1次追加位置必须是0
AppendObjectResult result1 = ossClient.appendObject(request1);

// 第2次追加 (在上次位置后追加)
AppendObjectRequest request2 = new AppendObjectRequest(
    bucketName,
    objectName,
    new ByteArrayInputStream(content2.getBytes())
);
request2.setPosition(result1.getNextPosition()); // 使用上次返回的位置
AppendObjectResult result2 = ossClient.appendObject(request2);

// 第3次追加
AppendObjectRequest request3 = new AppendObjectRequest(
    bucketName,
    objectName,
    new ByteArrayInputStream(content3.getBytes())
);
request3.setPosition(result2.getNextPosition());
ossClient.appendObject(request3);
```

**异常处理**:
| 异常 | 原因 | 解决方法 |
|------|------|---------|
| PositionNotEqualToLength | 追加位置≠文件当前长度 | 设置position=文件当前长度 |
| ObjectNotAppendable | 文件不是Appendable类型 | 删除旧文件重新创建 |

**限制**:
- Object大小 ≤5GB
- 不支持冷归档、深度冷归档
- 不支持上传回调

### 3.7 预签名URL上传

**适用场景**:
- **第三方临时授权上传**: 合作伙伴上传合同
- **用户上传头像**: 无需暴露密钥
- **分享文件**: 临时访问权限

**特点**:
- 预签名URL无需权限即可生成
- 第三方可使用URL在有效期内上传
- URL过期后需重新生成

**URL有效时长**:
- SDK生成: 最长7天
- STSToken生成: 最长43200秒(12小时)

**预签名URL格式**:
```
https://bucket.oss-cn-hangzhou.aliyuncs.com/object.txt?
x-oss-date=20250520T082728Z&
x-oss-expires=3600&
x-oss-signature=81f3d2e5eaa67c432291577ed20af3b3f60df05ab3cddedcdce168ef707f7ad0&
x-oss-credential=LTAI%2F20250520%2Fcn-hangzhou%2Foss%2Faliyun_v4_request
```

**生成预签名URL**:
```java
// 设置过期时间(1小时)
Date expiration = new Date(new Date().getTime() + 3600 * 1000);

// 生成PUT请求的预签名URL
GeneratePresignedUrlRequest request =
    new GeneratePresignedUrlRequest(bucketName, objectName, HttpMethod.PUT);
request.setExpiration(expiration);

// 可选: 设置请求头
Map<String, String> headers = new HashMap<>();
headers.put(OSSHeaders.STORAGE_CLASS, StorageClass.Standard.toString());
headers.put(OSSHeaders.CONTENT_TYPE, "text/plain");
request.setHeaders(headers);

URL signedUrl = ossClient.generatePresignedUrl(request);
```

**使用预签名URL上传**:
```java
// 使用预签名URL上传文件
HttpPut httpPut = new HttpPut(signedUrl.toString());
FileEntity entity = new FileEntity(new File(localFilePath));
httpPut.setEntity(entity);

// 设置相同的请求头(如果生成URL时设置了)
httpPut.setHeader(OSSHeaders.STORAGE_CLASS, "Standard");
httpPut.setHeader(OSSHeaders.CONTENT_TYPE, "text/plain");

CloseableHttpClient httpClient = HttpClients.createDefault();
CloseableHttpResponse response = httpClient.execute(httpPut);
```

**注意事项**:
1. 生成URL时设置了Header,使用时也必须传递相同Header
2. 预签名URL在有效期内可多次访问
3. 多次上传同名文件有覆盖风险
4. 需要配置Bucket CORS策略

**设置上传回调**:
```java
// 构造回调参数
String callbackBody = "{\"callbackUrl\":\"http://example.com/callback\","
    + "\"callbackBody\":\"bucket=${bucket}&object=${object}\"}";
String callbackBase64 = Base64.getEncoder().encodeToString(callbackBody.getBytes());

// 设置回调Header
headers.put(OSSHeaders.OSS_HEADER_CALLBACK, callbackBase64);
```

### 3.8 预签名URL分片上传

**适用场景**:
- 客户端无法集成OSS SDK
- 需要上传大文件(>5GB)
- 第三方应用临时授权上传

**上传流程**:
```
1. 客户端请求初始化上传 (文件名、大小、分片大小)
   ↓
2. 服务端初始化分片上传 → 获取UploadId
   ↓
3. 服务端为每个分片生成预签名URL
   ↓
4. 客户端并发上传分片 (按partNumber上传对应分片)
   ↓
5. 客户端通知服务端上传完成
   ↓
6. 服务端校验分片 → 合并分片
```

**服务端初始化并生成URL**:
```java
// 1. 初始化分片上传
InitiateMultipartUploadResult initiateResult =
    ossClient.initiateMultipartUpload(bucketName, key);
String uploadId = initiateResult.getUploadId();

// 2. 计算分片数量
long fileSize = 104857600; // 100MB
long partSize = 5242880;   // 5MB
int partCount = (int) Math.ceil((double) fileSize / partSize);

// 3. 为每个分片生成预签名URL
List<String> signedUrls = new ArrayList<>();
for (int i = 1; i <= partCount; i++) {
    URL url = ossClient.generatePresignedUrl(
        bucketName,
        key,
        HttpMethod.PUT,
        expiration,
        partId,
        uploadId,
        i
    );
    signedUrls.add(url.toString());
}

// 4. 返回给客户端
return {
  "uploadId": uploadId,
  "signedUrls": signedUrls
};
```

**客户端上传分片**:
```javascript
// 必须按URL中的partNumber上传对应分片
for (let i = 0; i < signedUrls.length; i++) {
  const partNumber = i + 1;
  const signedUrl = signedUrls[i];
  const partData = getPartData(partNumber); // 获取对应分片数据

  await fetch(signedUrl, {
    method: 'PUT',
    body: partData
  });
}

// 通知服务端合并
await fetch('/complete_upload', {
  method: 'POST',
  body: JSON.stringify({ uploadId, objectKey })
});
```

**服务端校验并合并**:
```java
// 1. 列举已上传分片
ListPartsRequest listPartsRequest = new ListPartsRequest(bucketName, key, uploadId);
PartListing partListing = ossClient.listParts(listPartsRequest);

// 2. 收集分片ETag
List<PartETag> partETags = new ArrayList<>();
for (PartSummary part : partListing.getParts()) {
    partETags.add(new PartETag(part.getPartNumber(), part.getETag()));
}

// 3. 合并分片
CompleteMultipartUploadRequest completeRequest =
    new CompleteMultipartUploadRequest(bucketName, key, uploadId, partETags);
CompleteMultipartUploadResult result =
    ossClient.completeMultipartUpload(completeRequest);
```

### 3.9 上传回调（Callback）

**适用场景**:
- 文件上传后需要通知业务服务器
- 需要更新数据库记录
- 需要触发其他业务逻辑

**典型应用**:
- 上传成功后更新用户头像URL
- 上传视频后触发转码任务
- 上传文档后触发审核流程

**回调流程**:
```
1. 客户端上传文件到OSS (携带Callback参数)
   ↓
2. OSS完成文件上传
   ↓
3. OSS主动POST请求到业务服务器 (callbackUrl)
   ↓
4. 业务服务器处理并返回结果
   ↓
5. OSS将结果返回给客户端
```

**支持的操作**:
- ✅ 简单上传 (PutObject)
- ✅ 表单上传 (PostObject)
- ✅ 完成分片上传 (CompleteMultipartUpload)
- ❌ 追加上传 (AppendObject)

**配置回调参数**:

### 3.10 上传进度条

**适用场景**:
- 上传大文件需要显示进度
- 提升用户体验
- 监控上传状态

**进度事件类型**:
| 事件类型 | 说明 |
|---------|------|
| TRANSFER_STARTED_EVENT | 传输开始 |
| REQUEST_CONTENT_LENGTH_EVENT | 获取到文件总大小 |
| REQUEST_BYTE_TRANSFER_EVENT | 数据传输中(每次触发) |
| TRANSFER_COMPLETED_EVENT | 传输完成 |
| TRANSFER_FAILED_EVENT | 传输失败 |

**实现进度监听**:
```java
public class UploadProgressListener implements ProgressListener {
    private long bytesWritten = 0;
    private long totalBytes = -1;

    @Override
    public void progressChanged(ProgressEvent progressEvent) {
        long bytes = progressEvent.getBytes();
        ProgressEventType eventType = progressEvent.getEventType();

        switch (eventType) {
            case TRANSFER_STARTED_EVENT:
                System.out.println("开始上传...");
                break;

            case REQUEST_CONTENT_LENGTH_EVENT:
                this.totalBytes = bytes;
                System.out.println("文件总大小: " + totalBytes + " 字节");
                break;

            case REQUEST_BYTE_TRANSFER_EVENT:
                this.bytesWritten += bytes;
                int percent = (int)(this.bytesWritten * 100.0 / this.totalBytes);
                System.out.println("已上传: " + percent + "% (" + bytesWritten + "/" + totalBytes + ")");
                break;

            case TRANSFER_COMPLETED_EVENT:
                System.out.println("上传成功! 总计: " + bytesWritten + " 字节");
                break;

            case TRANSFER_FAILED_EVENT:
                System.out.println("上传失败! 已上传: " + bytesWritten + " 字节");
                break;
        }
    }
}

// 使用进度监听
ossClient.putObject(
    new PutObjectRequest(bucketName, key, file)
        .withProgressListener(new UploadProgressListener())
);
```

## 四、上传策略选择

### 4.1 按文件大小选择

| 文件大小 | 推荐方式 | 理由 |
|---------|---------|------|
| <100KB | 简单上传 | 一次HTTP请求即可 |
| 100KB~5GB | 简单上传 / 表单上传 | 根据场景选择 |
| 5GB~100GB | 断点续传上传 | 支持断点续传,自动重试 |
| >100GB | 分片上传 + 并发 | 提高上传速度 |

### 4.2 按应用场景选择

| 场景 | 推荐方式 | 特点 |
|------|---------|------|
| Web端用户上传头像 | 表单上传 | 减轻服务器压力 |
| 移动端上传视频 | STS临时凭证 | 权限可控,安全性高 |
| 服务端批量上传 | 简单上传 | 实现简单,效率高 |
| 日志实时追加 | 追加上传 | 支持持续追加 |
| 大文件备份 | 断点续传上传 | 支持断点续传 |
| 第三方合作上传 | 预签名URL | 临时授权,安全 |

### 4.3 按性能要求选择

| 性能要求 | 策略 |
|---------|------|
| 最快上传速度 | 分片上传 + 高并发 + 合理分片大小 |
| 网络不稳定 | 断点续传上传 + Checkpoint文件 |
| 降低服务器压力 | 客户端直传 (STS或预签名URL) |
| 大量小文件 | 批量简单上传 + 并发控制 |

## 五、安全性考虑

### 5.1 防止文件覆盖

**场景**: 多用户上传同名文件,防止数据丢失

**解决方案**:

1. **开启版本控制**
   ```java
   // 被覆盖的文件保存为历史版本
   bucketVersioningConfiguration.setStatus("Enabled");
   ```

2. **禁止覆盖同名文件**
   ```java
   // 简单上传
   metadata.setHeader("x-oss-forbid-overwrite", "true");

   // 表单上传
   formData.append('x-oss-forbid-overwrite', 'true');
   ```

3. **使用UUID生成唯一文件名**
   ```java
   String objectKey = "avatar/" + UUID.randomUUID() + ".jpg";
   ```

### 5.2 文件类型限制

**Post Policy限制**:
```java
PolicyConditions policyConds = new PolicyConditions();

// 限制文件大小
policyConds.addConditionItem(
    PolicyConditions.COND_CONTENT_LENGTH_RANGE,
    0,
    10485760  // 最大10MB
```

**服务端校验**:
```java

    ossClient.deleteObject(bucketName, objectKey);

```

### 5.3 权限控制

**STS临时凭证**:

### 6.2 避免热点问题

**问题**: 使用顺序前缀命名(时间戳)导致请求集中在特定分区

**解决方案**:

### 6.3 减少PUT请求费用

**问题**: 直接上传深度冷归档文件产生高额PUT请求费用

```
2. 通过生命周期规则自动转换为深度冷归档
   ↓

```

**生命周期规则示例**:
```json

  "rules": [{
    "id": "rule1",
    "status": "Enabled",
    "filter": {"prefix": "archive/"},
    "transitions": [{
      "days": 30,
      "storageClass": "DeepColdArchive"
    }]
  }]
}
```
