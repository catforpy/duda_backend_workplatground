## 📚 表作用说明

### 核心业务表（5张）

#### 1. bucket_config - Bucket配置表 ⭐⭐⭐⭐⭐
**作用**: 存储Bucket的核心配置信息
**重要性**: ⭐⭐⭐⭐⭐ 最核心
**使用频率**: 极高（每次OSS操作都要查询）

**主要功能**:
- 存储Bucket基本信息（名称、区域、存储类型）
- 存储API密钥（加密后的access_key_id和access_key_secret）
- 存储用户关联信息（user_id、user_type、tenant_id）
- 存储Bucket功能开关（CDN、CORS、版本控制等）
- 存储配额限制（最大文件数、最大存储空间）

**核心字段**:
- `bucket_name`: Bucket名称（全局唯一）
- `access_key_id`: 阿里云AccessKey ID（AES加密）
- `access_key_secret`: 阿里云AccessKey Secret（AES加密）
- `user_id`: 所属用户ID
- `storage_type`: 存储类型（aliyun-oss/tencent-cos等）

---

#### 2. object_metadata - 对象元数据表 ⭐⭐⭐⭐⭐
**作用**: 存储文件/对象的元数据信息
**重要性**: ⭐⭐⭐⭐⭐ 最核心
**使用频率**: 极高（上传、下载、列表等操作都要查询）

**主要功能**:
- 记录每个文件的详细信息（文件名、大小、类型）
- 记录文件访问统计（上传时间、访问次数、下载次数）
- 记录文件存储属性（存储类型、ACL权限）
- 支持软删除（status字段）

**核心字段**:
- `bucket_name`: 所属Bucket
- `object_key`: 对象键（文件在OSS中的路径）
- `file_size`: 文件大小
- `content_type`: MIME类型
- `upload_time`: 上传时间
- `status`: 状态（active/deleted）

---

#### 3. upload_record - 上传记录表 ⭐⭐⭐⭐
**作用**: 记录文件上传操作的详细过程
**重要性**: ⭐⭐⭐⭐
**使用频率**: 高（每次上传都记录）

**主要功能**:
- 记录上传的开始和完成时间
- 记录上传方式（简单上传/分片上传）
- 记录客户端信息（IP、User-Agent、设备ID）
- 记录上传状态（成功/失败/进行中）

**核心字段**:
- `upload_id`: 上传任务ID
- `upload_status`: 上传状态
- `upload_method`: 上传方式（SIMPLE/MULTIPART/RESUMABLE）
- `start_time` / `complete_time`: 时间记录

---

#### 4. resume_upload_record - 断点续传记录表 ⭐⭐⭐
**作用**: 记录断点续传任务的状态
**重要性**: ⭐⭐⭐

**主要功能**:

**核心字段**:

## 🔍 字段重复分析

### ❌ 严重重复

#### 1. bucket_config 与 bucket_statistics - 大量配置字段重复

**重复字段清单**:
```sql
-- 这些字段在两个表中都存在 ❌
acl_type                  -- ACL类型
cors_enabled              -- 是否启用CORS
cors_config               -- CORS配置
referer_enabled           -- 是否启用防盗链
lifecycle_config          -- 生命周期配置
versioning_enabled        -- 是否启用版本控制
logging_config            -- 日志配置
website_config            -- 静态网站配置
worm_enabled              -- 是否启用WORM
replication_config        -- 跨区域复制配置
transfer_acceleration_enabled  -- 传输加速
access_monitor_enabled    -- 访问监控
domain_name               -- 自定义域名
cdn_enabled               -- 是否启用CDN
allowed_file_types        -- 允许的文件类型
blocked_file_types        -- 禁止的文件类型
max_file_size             -- 最大文件大小
max_file_count            -- 最大文件数
```

**问题分析**:
- `bucket_config` 是**配置表**，存储静态配置
- `bucket_statistics` 是**统计表**，存储动态统计数据
- 混在一起导致：
  - 数据冗余（一份配置存两份）
  - 更新困难（修改配置要同时改两个表）
  - 职责不清（不知道该查哪个表）

**解决方案**:
- ✅ **从 `bucket_statistics` 删除所有配置字段**
- ✅ **统计数据只保留**：文件数、存储大小、流量、费用等
- ✅ **需要配置时 JOIN `bucket_config` 表**

---

#### 2. bucket_config 与 bucket_statistics - 费用字段重复

**重复字段**:
```sql
-- bucket_statistics
| storage_cost | decimal(10,2) | 流量费用 |
| traffic_cost  | decimal(10,2) | 存储费用 |
| total_cost    | decimal(10,2) | 总费用 |

-- bucket_billing_record
| storage_fee | decimal(10,2) | 存储费用 |
| traffic_fee | decimal(10,2) | 流量费用 |
| request_fee | decimal(10,2) | 请求费用 |
| other_fee   | decimal(10,2) | 其他费用 |
| discount_fee| decimal(10,2) | 折扣金额 |
| total_fee   | decimal(10,2) | 总费用 |
```

**问题分析**:
- `bucket_statistics` 的费用是**实时估算值**
- `bucket_billing_record` 的费用是**正式账单**
- 字段名不统一（cost vs fee）

**解决方案**:
- ✅ `bucket_statistics` 保留费用字段（用于控制台实时展示）
- ✅ `bucket_billing_record` 保留费用字段（用于账单）
- ⚠️ 但要**统一字段命名规范**（建议都用fee）

---

### ⚠️ 轻度重复

#### 3. bucket_storage_log 与 bucket_statistics - 存储数据部分重复

**重复内容**:
```sql
-- bucket_storage_log (时间序列数据)
| standard_size    | bigint | 标准存储大小 |
| ia_size          | bigint | 低频存储大小 |
| archive_size     | bigint | 归档存储大小 |
| cold_archive_size| bigint | 冷归档存储大小 |

-- bucket_statistics (当前快照)
-- 通过 total_storage_size 一个字段汇总，不重复
```

**分析**:
- 这是**合理的设计**
- `bucket_storage_log` 是时间序列历史数据
- `bucket_statistics` 是当前快照
- 类似于交易明细 vs 账户余额

**结论**: ✅ **不需要修改**

## 🔪 过度设计与精简建议

### ❌ 过度设计

#### 1. file_statistics 表 - 功能重复 ⚠️

**问题**: 与 `bucket_storage_log` 和 `bucket_traffic_log` 功能重叠

**对比分析**:
```sql
-- file_statistics 的功能
| stat_date   | 统计日期          |
| total_files | 总文件数          |
| new_files   | 新增文件数        |  ← bucket_storage_log 也有
| deleted_files| 删除文件数       |  ← bucket_storage_log 也有
| upload_count | 上传数           |  ← bucket_traffic_log 也有
| download_count| 下载数          |  ← bucket_traffic_log 也有
| traffic_upload | 上传流量        |  ← bucket_traffic_log 也有
| traffic_download| 下载流量       |  ← bucket_traffic_log 也有
```

**建议**: ✅ **删除 `file_statistics` 表**

**理由**:
- `bucket_storage_log` 已经记录了存储变化
- `bucket_traffic_log` 已经记录了流量变化
- `file_statistics` 的数据可以通过聚合查询得到
- 避免维护三张表的数据一致性

---

#### 2. bucket_billing_config.balance 字段 - 位置不当 ⚠️

**问题**: 余额是实时数据，不应放在配置表

```sql
-- ❌ 当前设计
bucket_billing_config (配置表)
| balance | decimal(10,2) | 当前余额 |  ← 经常变化

-- 问题：
-- 1. 配置表应该不常变，但余额经常变化
-- 2. 每次充值/扣费都要更新这张表
-- 3. 无法记录余额变化历史
-- 4. 无法支持冻结金额、授信额度等复杂账户功能
```

**建议**: ✅ **创建独立的 `bucket_account` 表**

---

## 📝 每张表的详细修改建议

### 1. bucket_config（Bucket配置表）⭐⭐⭐⭐⭐

#### 修改建议

##### 修改1: 添加索引
```sql
-- ❌ 当前只有单列索引
-- ✅ 添加组合索引
CREATE INDEX idx_user_status ON bucket_config(user_id, is_deleted);
CREATE INDEX idx_user_type_status ON bucket_config(user_type, status, is_deleted);
CREATE INDEX idx_tenant_status ON bucket_config(tenant_id, is_deleted);

-- 理由：最常用的查询是"查用户未删除的Bucket"
```

##### 修改2: 缩短object_key相关字段
```sql
-- ❌ 当前：varchar(1024)
-- ✅ 修改为：varchar(512)

-- 理由：
-- 1. 1024字符会导致索引超长
-- 2. OSS的object_key一般不会超过512字符
-- 3. 如果确实需要长路径，可以用前缀索引
```
