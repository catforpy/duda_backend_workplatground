# 都达云台 (DudaNexus) 项目命名规范手册

## 1. 项目总览

### 1.1 项目基本信息
- **项目名称**：DudaNexus（都达云台）
- **项目代号**：DN
- **项目版本**：1.0.0-SNAPSHOT
- **项目描述**：都达网络科技有限公司微服务综合平台

### 1.2 命名原则
1. **语义化**：名称应清晰表达功能用途
2. **一致性**：同类资源采用统一命名模式
3. **可扩展性**：为未来扩展预留空间
4. **国际化**：采用英文命名，支持多语言

## 2. 模块命名规范

### 2.1 微服务模块命名
格式：`duda-{业务领域}-{功能}-service`

**示例：**
```
duda-auth-service          # 认证服务
duda-user-service          # 用户服务
duda-order-service         # 订单服务
duda-product-service       # 商品服务
duda-im-service            # 即时通讯服务
duda-live-service          # 直播服务
duda-social-service        # 社交服务
duda-payment-service       # 支付服务
duda-search-service        # 搜索服务
duda-notification-service  # 通知服务
```

### 2.2 API模块命名
格式：`duda-{业务领域}-{功能}-api`

**示例：**
```
duda-auth-api       # 认证API
duda-user-api       # 用户API
duda-order-api      # 订单API
duda-product-api    # 商品API
duda-im-api         # 即时通讯API
```

### 2.3 公共模块命名
格式：`duda-{组件类型}-{组件名称}`

**组件类型分类：**
- `common`：通用工具类
- `gateway`：网关
- `config`：配置中心
- `monitor`：监控
- `logging`：日志

**示例：**

## 3. 包名规范

### 3.1 基础包结构
```
com.duda.{service}
├── controller      # 控制器层
├── service         # 业务逻辑层
│   └── impl       # 业务实现
├── mapper          # 数据访问层
├── entity          # 实体类
│   ├── domain     # 领域对象
│   ├── dto        # 数据传输对象
│   └── vo         # 视图对象
├── config          # 配置类
├── common          # 公共类
│   ├── constant   # 常量
│   ├── enums      # 枚举
│   ├── exception  # 异常
│   └── utils      # 工具类
└── infrastructure  # 基础设施
    ├── mq         # 消息队列
    ├── cache      # 缓存
    └── storage    # 存储
```

### 3.2 包命名规则
- **全部小写**：包名全部使用小写字母
- **域名倒置**：使用公司域名倒置 `com.duda`
- **业务分隔**：使用点号分隔不同层级

## 4. 类命名规范

### 4.1 实体类命名
- **Entity（实体）**：与数据库表对应
  - 命名：`{业务名称}` 或 `{业务名称}Entity`
  - 示例：`User`, `Order`, `Product`

- **DTO（数据传输对象）**：跨服务传输
  - 命名：`{业务名称}DTO`
  - 示例：`UserDTO`, `OrderDTO`

- **VO（视图对象）**：返回前端
  - 命名：`{业务名称}VO`
  - 示例：`UserVO`, `OrderVO`

- **Query（查询对象）**：接收查询参数
  - 命名：`{业务名称}Query`
  - 示例：`UserQuery`, `OrderQuery`

- **Command（命令对象）**：接收命令参数
  - 命名：`{业务名称}Command`
  - 示例：`CreateUserCommand`, `UpdateOrderCommand`

### 4.2 Controller命名
格式：`{业务名称}Controller`

**示例：**
```java
UserController
OrderController
ProductController
AuthController
```

### 4.3 Service命名
- **接口**：`{业务名称}Service`
- **实现类**：`{业务名称}ServiceImpl`

**示例：**
```java
public interface UserService { }
public class UserServiceImpl implements UserService { }
```

### 4.4 Mapper命名
格式：`{业务名称}Mapper`

**示例：**
```java
UserMapper
OrderMapper
ProductMapper
```

### 4.5 配置类命名
格式：`{功能名称}Config`

**示例：**
```java
RedisConfig
WebMvcConfig
SecurityConfig
MybatisConfig
```

## 5. 方法命名规范

### 5.1 Controller方法
- **查询**：`get{资源名}`, `query{资源名}`, `list{资源名}`
- **创建**：`create{资源名}`, `add{资源名}`
- **更新**：`update{资源名}`, `modify{资源名}`
- **删除**：`delete{资源名}`, `remove{资源名}`

**示例：**
```java
@GetMapping("/user/{id}")
public Result<UserVO> getUser(@PathVariable Long id) { }

@GetMapping("/users")
public Result<List<UserVO>> listUsers(UserQuery query) { }

@PostMapping("/user")
public Result<Long> createUser(@RequestBody CreateUserCommand cmd) { }

@PutMapping("/user/{id}")
public Result<Boolean> updateUser(@PathVariable Long id, @RequestBody UpdateUserCommand cmd) { }

@DeleteMapping("/user/{id}")
public Result<Boolean> deleteUser(@PathVariable Long id) { }
```

### 5.2 Service方法
- **查询单个**：`get{业务名}ById`, `find{业务名}ById`
- **查询列表**：`list{业务名}`, `query{业务名}`
- **分页查询**：`page{业务名}`
- **保存**：`save{业务名}`
- **更新**：`update{业务名}`
- **删除**：`delete{业务名}`, `remove{业务名}`
- **判断是否存在**：`exists{业务名}`, `check{业务名}`
- **统计**：`count{业务名}`, `statistics{业务名}`

**示例：**
```java
User getUserById(Long id);
List<User> listUsers(UserQuery query);
PageResult<User> pageUsers(UserQuery query);
Long saveUser(User user);
Boolean updateUser(User user);
Boolean deleteUser(Long id);
Boolean existsUser(String username);
Integer countUsers();
```

## 6. 变量命名规范

### 6.1 基本原则
1. **驼峰命名法**：小驼峰（camelCase）
2. **见名知意**：变量名应清晰表达含义
3. **避免缩写**：除非是通用缩写
4. **布尔变量**：以 `is`, `has`, `can`, `should` 开头

### 6.2 变量命名示例
```java
// 正确示例
String userName;
Integer userAge;
Boolean isActive;
List<User> userList;
Map<Long, User> userMap;

// 错误示例
String n;    // 不明确
Integer ua;  // 缩写不清晰
Boolean flag; // 不明确
List<User> list; // 类型重复
```

### 6.3 常量命名
- **全大写**：使用全大写字母
- **下划线分隔**：使用下划线分隔单词

**示例：**
```java
public static final String DEFAULT_CHARSET = "UTF-8";
public static final Integer MAX_PAGE_SIZE = 100;
public static final Long CACHE_EXPIRE_TIME = 3600L;
```

## 7. 数据库命名规范

### 7.1 表命名
格式：`dn_{业务模块}_{业务功能}`

**规则：**
- **前缀**：统一使用 `dn_` (DudaNexus)
- **小写**：全部使用小写
- **下划线分隔**：使用下划线分隔单词

**示例：**
```sql
-- 用户模块
dn_user                    # 用户表
dn_user_profile            # 用户资料表
dn_user_relation           # 用户关系表

-- 订单模块
dn_order                   # 订单表
dn_order_item              # 订单明细表
dn_order_payment           # 订单支付表

-- 商品模块
dn_product                 # 商品表
dn_product_category        # 商品分类表
dn_product_sku             # 商品SKU表

-- IM模块
dn_im_message              # 消息表
dn_im_conversation         # 会话表
dn_im_group                # 群组表

-- 直播模块
dn_live_room               # 直播间表
dn_live_stream             # 流表
dn_live_gift               # 礼物表
```

### 7.2 字段命名
- **小写**：全部使用小写
- **下划线分隔**：使用下划线分隔单词
- **前缀标识**：避免关键字冲突

**通用字段：**
```sql
id              # 主键
create_time     # 创建时间
update_time     # 更新时间
create_by       # 创建人
update_by       # 更新人
is_deleted      # 是否删除
version         # 版本号（乐观锁）
tenant_id       # 租户ID
```

**示例：**
```sql
CREATE TABLE dn_user (
    id              BIGINT          NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    user_name       VARCHAR(50)     NOT NULL COMMENT '用户名',
    user_nick       VARCHAR(100)    NOT NULL COMMENT '昵称',
    user_email      VARCHAR(100)    COMMENT '邮箱',
    user_phone      VARCHAR(20)     COMMENT '手机号',
    user_avatar     VARCHAR(500)    COMMENT '头像URL',
    is_active       TINYINT         DEFAULT 1 COMMENT '是否激活',
    create_time     DATETIME        DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time     DATETIME        DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    is_deleted      TINYINT         DEFAULT 0 COMMENT '是否删除',
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_name (user_name),
    KEY idx_user_email (user_email),
    KEY idx_user_phone (user_phone)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 7.3 索引命名
- **主键索引**：`pk_{字段名}`
- **唯一索引**：`uk_{字段名}`
- **普通索引**：`idx_{字段名}`
- **联合索引**：`idx_{字段名1}_{字段名2}`

**示例：**
```sql
PRIMARY KEY (id)                          -- pk_id
UNIQUE KEY uk_user_name (user_name)       -- uk_user_name
KEY idx_user_email (user_email)           -- idx_user_email
KEY idx_user_phone_status (user_phone, is_active)  -- idx_user_phone_status
```

## 16. 版本号规范

### 16.1 版本号格式
格式：`{主版本}.{次版本}.{修订版}-{里程碑}`

**规则：**
- **主版本（Major）**：不兼容的API修改
- **次版本（Minor）**：向下兼容的功能性新增
- **修订版（Patch）**：向下兼容的问题修正
- **里程碑**：SNAPSHOT（快照）、RC（发布候选）、RELEASE（正式发布）

**示例：**
```
1.0.0-SNAPSHOT    # 开发中
1.0.0-RC1         # 发布候选1
1.0.0-RELEASE     # 正式发布
1.1.0-SNAPSHOT    # 新功能开发
2.0.0-SNAPSHOT    # 重大版本
```

## 21. 总结

本命名规范手册旨在为DudaNexus项目提供统一的命名标准，所有开发人员必须严格遵守。统一的命名规范将带来以下好处：

1. **提高代码可读性**：清晰的命名使代码更易理解
2. **降低维护成本**：统一的规范便于团队协作
3. **提升开发效率**：减少沟通成本，快速定位问题
4. **保证系统质量**：规范化的命名是高质量代码的基础

**重要提醒：**
- 所有新增代码必须符合本规范
- Code Review时重点检查命名规范
- 定期review和更新本规范，保持与时俱进
- 特殊情况需团队讨论决定，不可随意更改
