# 数据库设计规范手册

## 1. 数据库命名规范

### 1.1 库名命名
格式：`dn_{业务领域}`

**原则：**
- 使用小写字母
- 使用下划线分隔
- 语义清晰，见名知意
- 不超过20个字符

**示例：**
```sql
dn_auth          # 认证库
dn_user          # 用户库
dn_product       # 商品库
dn_order         # 订单库
dn_payment       # 支付库
dn_im            # 即时通讯库
dn_live          # 直播库
dn_social        # 社交库
dn_content       # 内容库
```

### 1.2 表名命名
格式：`dn_{模块}_{业务}`

**示例：**
```sql
-- 用户模块
dn_user                  # 用户表
dn_user_profile          # 用户资料表
dn_user_relation         # 用户关系表
dn_user_login_log        # 用户登录日志

-- 订单模块
dn_order                 # 订单表
dn_order_item            # 订单明细表
dn_order_payment         # 订单支付表
dn_order_log             # 订单日志表

-- 商品模块
dn_product               # 商品表
dn_product_category      # 商品分类表
dn_product_sku           # 商品SKU表
dn_product_spu           # 商品SPU表
```

## 2. 字段设计规范

### 2.1 数据类型选择

#### 2.1.1 整数类型
| 类型 | 字节 | 范围 | 用途 |
|------|------|------|------|
| TINYINT | 1 | -128~127 | 状态、标识 |
| SMALLINT | 2 | -32768~32767 | 小范围数量 |
| INT | 4 | -21亿~21亿 | 普通ID、数量 |
| BIGINT | 8 | 极大 | 主键ID、金额（分）|

**示例：**
```sql
id              BIGINT          NOT NULL AUTO_INCREMENT  -- 主键
status          TINYINT         DEFAULT 1                -- 状态
count           INT             DEFAULT 0                -- 数量
amount          BIGINT          DEFAULT 0                -- 金额（分）
```

#### 2.1.2 字符串类型
| 类型 | 用途 | 说明 |
|------|------|------|
| CHAR(1) | 单字符 | 固定长度，如性别、状态 |
| VARCHAR(N) | 变长字符串 | 用户名、标题、描述等 |
| TEXT | 长文本 | 内容、详情等 |

**示例：**
```sql
user_name       VARCHAR(50)     NOT NULL                -- 用户名
title           VARCHAR(200)    NOT NULL                -- 标题
content         TEXT            NULL                     -- 内容
```

#### 2.1.3 时间类型
| 类型 | 用途 |
|------|------|
| DATE | 日期（不含时间）|
| DATETIME | 日期时间 |
| TIMESTAMP | 时间戳（自动更新）|

**示例：**
```sql
create_time     DATETIME        DEFAULT CURRENT_TIMESTAMP
update_time     DATETIME        DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

#### 2.1.4 小数类型
| 类型 | 用途 | 示例 |
|------|------|------|
| DECIMAL(M,D) | 金额、精确小数 | DECIMAL(10,2) |

**示例：**
```sql
price           DECIMAL(10,2)   DEFAULT 0.00            -- 价格
```

### 2.2 通用字段设计

每个表必须包含以下通用字段：

```sql
CREATE TABLE dn_example (
    -- 主键
    id              BIGINT          NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 业务字段（根据实际业务添加）

    -- 审计字段
    create_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    create_by       BIGINT          NULL COMMENT '创建人ID',
    update_by       BIGINT          NULL COMMENT '更新人ID',

    -- 逻辑删除
    is_deleted      TINYINT         NOT NULL DEFAULT 0 COMMENT '是否删除：0否1是',

    -- 版本号（乐观锁）
    version         INT             NOT NULL DEFAULT 0 COMMENT '版本号',

    -- 租户ID（多租户场景）
    tenant_id       BIGINT          NOT NULL DEFAULT 0 COMMENT '租户ID',

    PRIMARY KEY (id),
    KEY idx_create_time (create_time),
    KEY idx_is_deleted (is_deleted)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='示例表';
```

### 2.3 字段命名规范

**命名规则：**
1. 使用小写字母
2. 使用下划线分隔单词
3. 见名知意
4. 布尔类型字段以 `is_` 开头

**示例：**
```sql
-- 正确示例
user_name       VARCHAR(50)     COMMENT '用户名'
user_email      VARCHAR(100)    COMMENT '邮箱'
is_active       TINYINT         COMMENT '是否激活'
is_deleted      TINYINT         COMMENT '是否删除'
create_time     DATETIME        COMMENT '创建时间'

-- 错误示例
userName        VARCHAR(50)     -- 应使用下划线
u_email         VARCHAR(100)    -- 缩写不清晰
flag            TINYINT         -- 语义不明确
```

### 2.4 字段约束

**NOT NULL约束：**
- 主键字段必须NOT NULL
- 重要业务字段建议NOT NULL
- 可选字段可以为NULL

**DEFAULT值：**
- 数量类型默认0
- 状态字段默认正常状态
- 时间字段默认CURRENT_TIMESTAMP
- 布尔字段默认0或1

**示例：**
```sql
status          TINYINT         NOT NULL DEFAULT 1 COMMENT '状态：1正常0禁用'
count           INT             NOT NULL DEFAULT 0 COMMENT '数量'
is_active       TINYINT         NOT NULL DEFAULT 1 COMMENT '是否激活'
create_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'
```

### 2.5 字段注释

**强制要求：**
- 所有字段必须添加COMMENT
- 注释要清晰描述字段含义
- 枚举值要在注释中说明

**示例：**
```sql
status          TINYINT         NOT NULL DEFAULT 1 COMMENT '状态：1正常2禁用3删除'
order_type      TINYINT         NOT NULL COMMENT '订单类型：1普通订单2秒杀订单3拼团订单'
pay_type        TINYINT         NULL COMMENT '支付方式：1微信2支付宝3余额'
```

## 3. 索引设计规范

### 3.1 索引命名规范

**命名规则：**
- 主键索引：`pk_{字段名}`
- 唯一索引：`uk_{字段名}`
- 普通索引：`idx_{字段名}`
- 联合索引：`idx_{字段1}_{字段2}_{字段3}`

**示例：**
```sql
PRIMARY KEY (id)                          -- pk_id
UNIQUE KEY uk_user_name (user_name)       -- uk_user_name
KEY idx_user_email (user_email)           -- idx_user_email
KEY idx_user_phone_status (user_phone, is_active)  -- idx_user_phone_status
```

### 3.2 索引设计原则

**应该建索引：**
1. 频繁作为WHERE条件的字段
2. 经常ORDER BY、GROUP BY的字段
3. 外键字段
4. 频繁用于JOIN的字段

**不应该建索引：**
1. 区分度不高的字段（如性别）
2. 频繁更新的字段
3. 很少查询的字段
4. 大文本字段

### 3.3 联合索引设计

**最左前缀原则：**
- 将区分度高的字段放在前面
- 将经常单独查询的字段放在前面
- 考虑字段大小，小的放前面

**示例：**
```sql
-- 查询场景：根据用户ID和状态查询
-- SELECT * FROM dn_order WHERE user_id = ? AND status = ?;
KEY idx_user_id_status (user_id, status)

-- 查询场景：根据用户ID查询，按创建时间倒序
-- SELECT * FROM dn_order WHERE user_id = ? ORDER BY create_time DESC;
KEY idx_user_id_create_time (user_id, create_time)
```

### 3.4 索引优化建议

**覆盖索引：**
```sql
-- 查询只需要id和status
-- SELECT id, status FROM dn_order WHERE user_id = ?;
KEY idx_user_id_id_status (user_id, id, status)
```

**避免索引失效：**
1. 避免在索引列上使用函数
2. 避免隐式类型转换
3. 避免前缀模糊查询（LIKE '%abc'）
4. 避免OR连接（除非所有字段都有索引）

## 4. 表设计规范

### 4.1 存储引擎

**统一使用InnoDB：**
- 支持事务
- 支持外键
- 支持行锁
- 崩溃恢复能力好

```sql
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.2 字符集

**统一使用utf8mb4：**
- 支持emoji表情
- 完整支持UTF-8
- 向后兼容utf8

```sql
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 4.3 主键设计

**原则：**
1. 必须有主键
2. 使用BIGINT自增
3. 单列主键，不使用联合主键
4. 尽量不用业务字段做主键

**示例：**
```sql
id              BIGINT          NOT NULL AUTO_INCREMENT COMMENT '主键ID',
PRIMARY KEY (id)
```

### 4.4 外键设计

**建议：**
- 不建议使用物理外键
- 使用逻辑外键（应用层控制）
- 避免外键影响性能

**示例：**
```sql
-- 物理外键（不推荐）
FOREIGN KEY (user_id) REFERENCES dn_user(id)

-- 逻辑外键（推荐）
-- 在应用层保证关联数据的完整性
```

## 5. 分库分表策略

### 5.1 分库策略

**按业务域分库：**
```
dn_user          # 用户中心库
dn_order         # 订单库
dn_product       # 商品库
dn_payment       # 支付库
```

### 5.2 分表策略

**水平分表：**
- 数据量超过500万考虑分表
- 可按ID取模、时间、哈希等方式

**示例：**
```sql
-- 按月分表
dn_order_202401
dn_order_202402
dn_order_202403

-- 按ID取模分表（0-9）
dn_order_0
dn_order_1
dn_order_2
...
```

## 6. 数据一致性保证

### 6.1 读写分离

**使用ShardingSphere实现：**
```yaml
spring:
  shardingsphere:
    datasource:
      names: master,slave0,slave1
      master:
        jdbc-url: jdbc:mysql://master:3306/dn_user
      slave0:
        jdbc-url: jdbc:mysql://slave0:3306/dn_user
      slave1:
        jdbc-url: jdbc:mysql://slave1:3306/dn_user
    rules:
      readwrite-splitting:
        data-sources:
          dn_user:
            write-data-source-name: master
            read-data-source-names: slave0,slave1
            load-balancer-name: round-robin
```

### 6.2 分布式事务

**使用Seata保证：**
```java
@GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
public void createOrder(Order order) {
    // 跨多个数据库的操作
    orderDao.insert(order);
    productDao.deductStock(order.getProductId(), order.getCount());
}
```

### 6.3 数据同步方案

**基于MQ的最终一致性：**
1. 本地事务 + 消息表
2. 发送MQ消息
3. 消费者处理消息
4. 重试机制

## 8. SQL编写规范

### 8.1 命名规范

```sql
-- 表别名使用有意义的缩写
SELECT u.user_name, o.order_no
FROM dn_user u
INNER JOIN dn_order o ON u.id = o.user_id;

-- 避免使用SELECT *
SELECT user_name, user_email, user_phone
FROM dn_user
WHERE id = 1;
```

### 8.2 性能优化

```sql
-- 避免在WHERE中使用函数
-- 错误示例
SELECT * FROM dn_user WHERE DATE(create_time) = '2024-03-07';

-- 正确示例
SELECT * FROM dn_user WHERE create_time >= '2024-03-07 00:00:00' AND create_time <= '2024-03-07 23:59:59';

-- 使用LIMIT限制返回行数
SELECT * FROM dn_order WHERE user_id = 1 LIMIT 100;

-- 避免大事务
-- 错误示例：在一个事务中处理大量数据
-- 正确示例：分批处理
```

### 8.3 分页查询优化

```sql
-- 传统分页（数据量大时性能差）
SELECT * FROM dn_order ORDER BY id LIMIT 1000000, 10;

-- 优化方案1：使用上次查询的最大ID
SELECT * FROM dn_order WHERE id > last_max_id ORDER BY id LIMIT 10;

-- 优化方案2：延迟关联
SELECT o.* FROM dn_order o
INNER JOIN (SELECT id FROM dn_order ORDER BY id LIMIT 1000000, 10) t
ON o.id = t.id;
```

## 12. 常见问题与解决方案

### 12.1 死锁问题

**原因：**
- 多个事务相互等待对方释放锁
- 事务执行时间过长

**解决方案：**
1. 减小事务范围
2. 统一访问顺序
3. 添加重试机制
4. 使用乐观锁

### 12.2 主从延迟

**原因：**
- 从库性能不足
- 大事务导致

**解决方案：**
1. 优化从库性能
2. 拆分大事务
3. 使用半同步复制

### 12.3 连接池耗尽

**原因：**
- 连接未释放
- 连接池配置过小

**解决方案：**
